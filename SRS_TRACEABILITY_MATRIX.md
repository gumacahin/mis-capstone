# SRS Traceability Matrix

Date: 2026-06-28

## Purpose

This matrix reconciles the original ReadySET SRS copied into `docs/` with the
current Django/React implementation.

The original SRS frames WSIDT as a web-based task-management and
productivity-tracking tool for the UPOU community, with usefulness evaluated for
UPOU faculty and staff. The current implementation preserves the task system as
the operational foundation, but freezes the capstone contribution around the
planner-first question from Crisanto's study:

```text
What should I do today?
```

The project should therefore be evaluated as a planner-first MVP, not as an
attempt to fully recreate Todoist, Trello, Google Calendar, or a complete
enterprise productivity suite.

## Gen UI Contribution Mapping

The original SRS does not explicitly require generative UI. The capstone can
defend generative UI as the design strategy for satisfying the most important
planning-related SRS items:

| SRS pressure                        | Static UI limitation                                                          | Current Gen UI / JIT UI response                                                                        |
| ----------------------------------- | ----------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------- |
| F-12 suggested tasks                | A generic task list still leaves the user deciding what matters now.          | `/today` generates an ordered plan with reasons and action buttons.                                     |
| F-21 personal scheduling            | Calendar/task views show dates but do not negotiate energy or available time. | Planner check-in captures available minutes and focus mode, then fits suggestions to that context.      |
| F-41 productivity analytics         | Reports are retrospective and do not necessarily help the user act today.     | Planner feedback and action rates evaluate whether recommendations increase confidence and helpfulness. |
| F-42 energy tracking                | Energy logs alone do not change the planning interface.                       | Low-energy, limited-time, and overdue-triage UI schemas select different planner surfaces.              |
| Voice/conversational interface note | Free-form assistant control could create unsafe database mutations.           | Typed operations and a fixed component registry allow future chat/voice without raw DB access.          |

## Capstone Completion Boundary

For the current capstone, "done" should mean:

- The original SRS is preserved as the product baseline.
- The task-management implementation is sufficient as a system of record.
- `/today` demonstrates the planner-first reinterpretation of suggested tasks.
- Energy, available time, focus mode, task signals, reasons, and planner
  feedback are present.
- Gen UI is demonstrated as safe just-in-time UI through a component registry
  and UI schema, not arbitrary generated frontend code.
- Admin/evaluation reporting stays aggregate and privacy-preserving.
- Optional integrations, group scheduling, motivation features, and full
  conversational/voice UI are explicitly deferred.

## Paper-Ready Summary

```text
The original SRS defined WSIDT as a web-based task-management and
productivity-tracking tool for the UPOU community. The implemented system
preserves that foundation through task, project, label, scheduling,
notification, productivity, and administrative features. However, the capstone
narrows the main contribution around the planning problem raised by Crisanto's
study: helping faculty and staff decide what to do today. The Release 1.0
suggested-tasks requirement is therefore expanded into a planner-first daily
workflow that uses task signals, energy level, available time, focus mode, and
feedback to present explainable recommendations. Generative UI is introduced as
a safe just-in-time interaction layer: the system selects from registered
planner components and calls typed backend operations, while Django remains the
system of record for validation, permissions, and privacy-preserving reporting.
```
